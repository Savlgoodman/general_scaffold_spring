# 认证架构文档

## 概述

本系统采用 JWT Token 实现无状态认证，支持 Access Token 和 Refresh Token 双令牌机制。

## 技术选型

| 组件 | 技术 | 说明 |
|------|------|------|
| JWT | java-jwt 4.4.0 | Token 签发与验证 |
| 验证码 | wf-captcha | 图形验证码生成 |
| 密码加密 | BCryptPasswordEncoder | Spring Security 内置 |
| 缓存 | Redis | Token 黑名单、验证码、登录失败计数 |

## 核心组件

### 1. SecurityConfig
**位置**: `config/SecurityConfig.java`

- 配置 Spring Security 过滤器链
- 放行路径：`/api/admin/auth/**`（登录、注册、验证码、刷新Token）
- 其他路径需认证
- 集成 `JwtAuthenticationFilter`

### 2. JwtTokenProvider
**位置**: `security/JwtTokenProvider.java`

职责：
- `generateAccessToken(userId, username)` - 生成 Access Token（默认30分钟）
- `generateRefreshToken(userId, username)` - 生成 Refresh Token（默认7天）
- `verifyToken(token)` - 验证 Token 有效性
- `addToBlacklist(token)` - 将 Token 加入黑名单
- `isInBlacklist(token)` - 检查 Token 是否在黑名单

存储结构：
- Access Token Claim: `userId`, `username`, `tokenType`
- Redis 黑名单 Key: `token:blacklist:{token}`

### 3. JwtAuthenticationFilter
**位置**: `security/JwtAuthenticationFilter.java`

- 继承 `OncePerRequestFilter`
- 从请求头 `Authorization: Bearer {token}` 提取 Token
- 验证 Token 类型（仅处理 Access Token）
- 检查 Token 黑名单
- 验证通过后设置 `SecurityContextHolder` 认证信息

### 4. AdminUserService & AdminUserDetails
**位置**: `service/AdminUserService.java` + `service/impl/AdminUserServiceImpl.java`

- `AdminUserService`：用户服务接口，定义 `loadUserByUsername`、`loadUserById`、`getUserById` 方法
- `AdminUserDetails`（内部类）：实现 `UserDetails` 接口，封装用户信息
- 职责：用户认证信息加载，是 Spring Security 与业务系统的桥接层

### 5. AuthCaptchaUtil
**位置**: `util/AuthCaptchaUtil.java`

职责：
- `generate(width, height, len, type, redisTemplate)` - 生成验证码
  - 支持类型：`png`(默认)、`gif`、`arithmetic`、`chinese`、`chineseGif`
  - 验证码存入 Redis，Key = `captcha:{uuid}`，TTL 5分钟
  - 返回 `CaptchaResult(captchaKey, captchaImage, type)`
- `verify(captchaKey, captchaCode, redisTemplate)` - 校验后自动删除
- `delete(captchaKey, redisTemplate)` - 删除验证码

### 6. AuthService
**位置**: `service/AuthService.java`

接口定义：
```java
CaptchaVO generateCaptcha();
LoginVO login(LoginDTO loginDTO);
LoginVO register(RegisterDTO registerDTO);
LoginVO refreshToken(RefreshTokenDTO refreshTokenDTO);
void logout(String accessToken);
```

### 7. AuthServiceImpl
**位置**: `service/impl/AuthServiceImpl.java`

核心逻辑：

**登录流程**：
1. 检查账户是否被锁定（Redis `login:fail:{username}` 计数 ≥ 5）
2. 校验图形验证码
3. 使用 `AuthenticationManager` 验证用户名密码
4. 清除登录失败计数
5. 生成 Access Token + Refresh Token
6. 记录登录日志到 `admin_login_log` 表
7. 返回 `LoginVO`

**登录安全**：
- Redis Key: `login:fail:{username}`
- 失败计数 ≥ 5 次，锁定 15 分钟
- 每次失败递增计数并重置TTL

**注册流程**：
1. 检查用户名是否已存在
2. BCrypt 加密密码
3. 插入 `admin_user` 表
4. 生成 Token
5. 记录登录日志

**刷新Token**：
1. 验证 Refresh Token 有效性
2. 检查是否在黑名单
3. 将旧 Refresh Token 加入黑名单
4. 生成新的 Access Token + Refresh Token

**登出**：
1. 解析 Bearer Token
2. 加入黑名单

### 8. AuthController
**位置**: `controller/AuthController.java`

接口：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/admin/auth/captcha` | 获取验证码 |
| POST | `/api/admin/auth/login` | 用户登录 |
| POST | `/api/admin/auth/register` | 用户注册 |
| POST | `/api/admin/auth/refresh` | 刷新Token |
| POST | `/api/admin/auth/logout` | 登出 |
| GET | `/api/admin/auth/me` | 获取当前用户 |

## 数据库表

### admin_user
管理员用户表

### admin_login_log
登录日志表

字段：`username`, `status`(success/failed/locked), `ip`, `userAgent`, `message`

## Redis Key 规范

| Key Pattern | 说明 | TTL |
|-------------|------|-----|
| `captcha:{uuid}` | 验证码 | 5分钟 |
| `login:fail:{username}` | 登录失败计数 | 15分钟 |
| `token:blacklist:{token}` | Token黑名单 | Token剩余有效期 |

## 配置项

```yaml
jwt:
  secret: ${jwt.secret}
  access-expiration: 1800000    # 30分钟
  refresh-expiration: 604800000  # 7天
```

## 经验教训

### 1. MVC 分层的重要性

**问题**：最初将验证码生成的参数（width, height, len, type）暴露在 Controller 的 URL 参数中。

**正确做法**：
- Controller：只负责接收请求、调用 service、返回结果
- Service：处理所有业务逻辑，包括验证码生成的参数配置
- Util：只负责纯算法（如验证码图片生成），不涉及业务

**教训**：业务配置应该由 service 层决定，而不是让前端随意控制。

### 2. 类名命名避免冲突

**问题**：自定义 `CaptchaUtil` 与 `com.wf.captcha.utils.CaptchaUtil` 类名冲突。

**解决**：重命名为 `AuthCaptchaUtil`。

**教训**：工具类命名应该带有业务前缀，避免与第三方库冲突。

### 3. UserDetails 归属问题

**问题**：最初将 `AdminUserDetails` 放在 `security/` 包下。

**正确做法**：`AdminUserDetails` 是用户服务的内部实现细节，应该放在 `AdminUserServiceImpl` 内部作为内部类。

**教训**：UserDetails 是 Spring Security 与业务系统的桥接，不应该独立存在。

### 4. Redis Key 集中管理

**问题**：早期 Redis Key 散落在各处（`captcha:`、`login:fail:`、`token:blacklist:`）。

**解决**：创建 `RedisKeys` 枚举类集中管理所有 Redis Key 前缀。

**好处**：
- 便于维护和修改
- 避免拼写错误
- 统一规范

### 5. 验证码架构流程

```
前端请求 /captcha
    ↓
AuthController.getCaptcha()
    ↓
AuthService.generateCaptcha()
    ↓
AuthCaptchaUtil.generate(redisTemplate) → 生成图片+存Redis
    ↓
返回 CaptchaVO { captchaKey, captchaImage }
```

**教训**：清晰的调用链便于调试和问题排查。

### 6. 第三方库版本注意

**问题**：`java-jwt:0.12.6` 版本不存在。

**解决**：使用正确版本 `4.4.0`。

**教训**：使用开源库时要确认版本号，可通过 Maven Central 查询。

### 7. 内部类使用规范

**情况**：在 `AdminUserServiceImpl` 内部定义 `AdminUserDetails` 内部类。

**引用方式**：`AdminUserServiceImpl.AdminUserDetails`

**教训**：当一个类只被另一个类紧密关联使用时，可以作为内部类处理。
